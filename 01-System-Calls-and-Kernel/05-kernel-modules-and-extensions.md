# Kernel Modules and Extensions

## What Are Loadable Kernel Modules?

A **Loadable Kernel Module (LKM)** is a piece of code that can be inserted into the running Linux kernel **without rebooting**. Once loaded, it runs in kernel mode (Ring 0) with full access to kernel data structures and hardware.

The most common use case: **device drivers**. When you plug in a USB device, the kernel automatically loads the appropriate module to talk to that device. When you remove it, the module can be unloaded.

```
KERNEL WITH MODULES
=====================

  +--------------------------------------------------+
  |               LINUX KERNEL (vmlinuz)              |
  |                                                   |
  |  Core: scheduler, memory manager, VFS, networking |
  |                                                   |
  |  +----------+  +----------+  +----------+        |
  |  | ext4.ko  |  | nvidia.ko|  | wireguard|        |
  |  | (loaded) |  | (loaded) |  |   .ko    |        |
  |  +----------+  +----------+  | (loaded) |        |
  |                               +----------+        |
  |                                                   |
  |  Modules share the kernel address space           |
  |  They can call kernel functions directly           |
  +--------------------------------------------------+
        ^               ^              ^
        |               |              |
     insmod          insmod         insmod
     (at boot)     (on demand)   (by admin)
```

## Why Modules Exist

Without modules, you'd have two bad options:

1. **Compile everything into the kernel** -- The kernel binary becomes enormous, wasting memory on drivers for hardware you don't have. Changing any driver requires recompiling and rebooting.

2. **Leave things out** -- Then you can't use hardware that wasn't anticipated when the kernel was compiled.

Modules solve this by allowing **on-demand loading**:

| Approach | Binary Size | Supports New Hardware? | Requires Reboot? |
|----------|-----------|----------------------|-----------------|
| Everything built-in | Large (100MB+) | No -- must recompile | Yes |
| Minimal kernel + modules | Small (~10MB) | Yes -- load new .ko | No |
| No drivers at all | Tiny | No | N/A |

A typical Linux distribution ships with **thousands** of modules in `/lib/modules/<kernel-version>/` and loads only the ones needed for the current hardware. Run `find /lib/modules/$(uname -r) -name "*.ko*" | wc -l` to see how many are available on your system.

## Module Management Commands

### The Essential Commands

| Command | Purpose | Example |
|---------|---------|---------|
| `lsmod` | List currently loaded modules | `lsmod \| grep nvidia` |
| `insmod` | Insert a module (low-level, no dependency handling) | `insmod ./mydriver.ko` |
| `rmmod` | Remove a loaded module | `rmmod mydriver` |
| `modprobe` | Smart insert/remove (handles dependencies) | `modprobe wireguard` |
| `modinfo` | Show module information (author, description, params) | `modinfo ext4` |
| `depmod` | Regenerate module dependency database | `depmod -a` |

### `modprobe` vs `insmod`

Always prefer `modprobe` over `insmod`:

```
MODULE DEPENDENCY EXAMPLE
==========================

  Module: usb_storage.ko
  Depends on: scsi_mod.ko, usb_core.ko

  With insmod (manual):              With modprobe (automatic):
  
  insmod scsi_mod.ko                 modprobe usb_storage
  insmod usb_core.ko                 # automatically loads
  insmod usb_storage.ko              # scsi_mod and usb_core first
  
  (You must know and load            (Reads dependencies from
   dependencies yourself)             modules.dep, loads them
                                      in correct order)
```

`modprobe` reads `/lib/modules/<version>/modules.dep` (generated by `depmod`) to figure out the dependency chain.

## Built-in vs Module: Tradeoffs

| Factor | Built-in (compiled in) | Module (.ko file) |
|--------|----------------------|-------------------|
| **Available at boot** | Yes -- always present | Only if loaded early (initramfs) or after boot |
| **Memory usage** | Always in RAM | Only in RAM when loaded |
| **Can be unloaded?** | No | Yes (if not in use) |
| **Update without reboot?** | No | Yes -- rmmod old, insmod new |
| **Boot-critical?** | Required for root filesystem driver, boot console | Everything else can be a module |
| **Performance** | Very slightly faster (no module lookup overhead) | Negligible difference in practice |

**Rule of thumb**: Build your root filesystem driver and essential boot hardware support into the kernel. Everything else can be a module.

The **initramfs** (initial RAM filesystem) is a clever solution for the chicken-and-egg problem: the kernel needs the filesystem driver to read modules from disk, but the filesystem driver might itself be a module. The initramfs is a small archive loaded into RAM by the bootloader, containing just enough modules and tools to mount the real root filesystem.

```
BOOT SEQUENCE WITH INITRAMFS
==============================

  Bootloader (GRUB)
       |
       | loads vmlinuz + initramfs into RAM
       v
  Kernel starts
       |
       | mounts initramfs as temporary root "/"
       v
  initramfs scripts run
       |
       | modprobe ext4       (load filesystem driver)
       | modprobe ahci       (load disk controller driver)
       | mount /dev/sda1 /   (mount real root)
       v
  pivot_root to real filesystem
       |
       | systemd/init starts
       | loads remaining modules on demand
       v
  System fully booted
```

## Security Implications

Modules run in **Ring 0** with full kernel privileges. This has serious security implications:

```
SECURITY RISKS OF KERNEL MODULES
==================================

  +------------------+
  | Loaded Module    |   Has access to:
  | (Ring 0)         |   - ALL physical memory
  +------------------+   - ALL kernel data structures
                         - ALL hardware I/O ports
                         - Can disable security mechanisms
                         - Can hide processes (rootkits)
                         - Can intercept syscalls
                         - Can log keystrokes
```

**Mitigations**:

| Mechanism | What It Does |
|-----------|-------------|
| **Module signing** | Kernel verifies cryptographic signature before loading. Unsigned modules rejected on Secure Boot systems. |
| **`CONFIG_MODULE_SIG_FORCE`** | Kernel config that requires ALL modules to be signed. |
| **Lockdown mode** | Restricts what even root can do: no unsigned modules, no `/dev/mem` access, no hibernation (which could be used to modify kernel memory). |
| **Disabling module loading** | `echo 1 > /proc/sys/kernel/modules_disabled` permanently prevents loading new modules (until reboot). |
| **SELinux/AppArmor** | Can restrict which processes are allowed to load modules. |

In production security-sensitive systems (financial, government), it's common to disable module loading entirely after boot.

## eBPF: The Modern Kernel Extension Mechanism

**eBPF** (extended Berkeley Packet Filter) is a revolutionary technology that lets you run sandboxed programs inside the kernel **without writing a kernel module**:

```
KERNEL MODULES vs eBPF
========================

  Traditional Module:                   eBPF Program:
  
  +------------------+                 +------------------+
  | mymodule.ko      |                 | myprog.bpf.o     |
  | Written in C     |                 | Written in C     |
  | Full kernel API  |                 | Restricted API   |
  | No verification  |                 | Verified by BPF  |
  | Can crash kernel |                 |   verifier       |
  | Needs root       |                 | Cannot crash     |
  | Hard to deploy   |                 |   kernel         |
  +------------------+                 | Easier to deploy |
                                       +------------------+
```

| Aspect | Kernel Module | eBPF |
|--------|-------------|------|
| **Safety** | Can crash the kernel | Verified for safety before loading |
| **API access** | Full kernel API | Limited, safe helper functions |
| **Loops** | Unrestricted | Must be bounded (verifier checks) |
| **Memory access** | Direct pointer dereference | Bounds-checked by verifier |
| **Use cases** | Device drivers, filesystems | Observability, networking, security |
| **Deployment** | Compile per-kernel, insmod | Portable across kernel versions (CO-RE) |

### What eBPF Is Used For

- **Observability**: Tools like `bpftrace`, Cilium's Hubble, and Pixie use eBPF to trace syscalls, function calls, and network packets with near-zero overhead
- **Networking**: Cilium replaces `iptables` with eBPF programs for Kubernetes networking, handling millions of packets/sec
- **Security**: Falco and Tracee use eBPF to detect suspicious syscall patterns (container escapes, privilege escalation) in real time
- **Profiling**: `perf` and BCC tools use eBPF to profile CPU usage, memory allocation, and I/O latency

eBPF is arguably the most important Linux kernel technology of the past decade. It lets you extend kernel behavior safely, without the risks of kernel modules.

## How This Connects to Device Driver Loading

When you plug in new hardware (say, a USB WiFi adapter):

```
AUTOMATIC MODULE LOADING (udev)
=================================

  1. Hardware connected
       |
       v
  2. Kernel detects new device via bus enumeration
     Emits uevent with device info (vendor ID, product ID)
       |
       v
  3. udev daemon receives the uevent
     Matches against rules in /etc/udev/rules.d/
       |
       v
  4. udev calls: modprobe <driver-name>
     (driver name determined from device ID -> module alias mapping)
       |
       v
  5. modprobe loads the .ko file + dependencies
       |
       v
  6. Driver's probe() function runs
     - Initializes the hardware
     - Registers with appropriate subsystem (network, block, etc.)
       |
       v
  7. Device is ready to use
     (network interface appears, block device appears, etc.)
```

The mapping from device IDs to module names lives in the `modules.alias` file, generated by `depmod` from `MODULE_DEVICE_TABLE` entries in each driver's source code.

## Real-World Connection

- **NVIDIA GPU drivers** are distributed as kernel modules. This is why NVIDIA driver updates sometimes break after a kernel upgrade -- the module was compiled for the old kernel's internal API. DKMS (Dynamic Kernel Module Support) automates recompilation of third-party modules when the kernel is updated.

- **Container security**: In Kubernetes, tools like Falco use eBPF to monitor every syscall in every container without modifying the containers themselves. This is the foundation of runtime security monitoring in cloud-native environments.

- **Cloud provider customization**: AWS, Google Cloud, and Azure all run custom kernel modules for their virtual hardware (virtio drivers for networking and storage). When you boot an EC2 instance, the kernel loads `virtio_net`, `virtio_blk`, and `nvme` modules to talk to the hypervisor's virtual devices.

- **WireGuard VPN** started as an out-of-tree kernel module before being merged into the mainline kernel (5.6). This is the typical lifecycle: prototype as a module, prove the design, then merge upstream.

## Interview Angle

**Q: What is a kernel module and why would you use one instead of compiling functionality into the kernel?**

A: A kernel module is a piece of code that can be loaded into or removed from a running kernel without rebooting. You'd use a module instead of building in when: (1) the functionality isn't needed at boot time, (2) you want to save memory by only loading drivers for hardware that's actually present, (3) you need to update a driver without rebooting (important for server uptime), or (4) you're developing a driver and want fast iteration. You'd compile something in when it's needed for booting (root filesystem driver) or when the slight overhead of module loading is unacceptable. In practice, modern distributions ship almost everything as modules.

**Q: What are the security risks of kernel modules, and how do you mitigate them?**

A: Kernel modules run in Ring 0 with full hardware access, so a malicious module can do anything: read all memory, install rootkits, intercept syscalls, log keystrokes. Mitigations include: module signing (the kernel verifies a cryptographic signature before loading, required with Secure Boot), lockdown mode (restricts root's ability to modify the running kernel), disabling module loading after boot (`/proc/sys/kernel/modules_disabled`), and using MAC systems like SELinux to restrict which processes can load modules. For maximum security, some environments compile everything needed into the kernel and disable module loading entirely.

**Q: What is eBPF and why is it significant?**

A: eBPF lets you run sandboxed programs inside the kernel without writing a kernel module. Before loading, the BPF verifier checks the program for safety: no unbounded loops, no invalid memory access, no kernel crashes. This makes it safe for production use. It's significant because it enables kernel-level observability (tracing syscalls, network packets), high-performance networking (Cilium replaces iptables for Kubernetes), and runtime security monitoring (detecting suspicious behavior) -- all without the risks and deployment complexity of kernel modules. It's often described as what JavaScript is to the browser: a safe way to extend the platform without modifying its core.
